| やりたいこと | 手段 |
|---|---|
| プロジェクトに指示を常駐させる | CLAUDE.md を作る → `/init` |
| 出力を短くする | `/genshijin` |

## CLAUDE.md を設計する

> [!info] なぜ CLAUDE.md か
> セッションをまたいで有効な指示ファイル。モデルへの口頭説明を毎回繰り返す代わりに、プロジェクトの文脈・規約・ワークフローを一箇所に集約する。`/init` で自動生成できる。

## トークンを効率化する

### .claudeignore

不要ファイルをコンテキストから除外する。プロジェクトルートに配置。

```
node_modules/
.git/
dist/
*.log
*.lock
coverage/
```

### コンテキスト管理

```
/compact    # 圧縮（穏やか）
/clear      # リセット（完全）
```

長いセッションは途中リセット＋新セッション継続がコスト・品質ともに良い。

### MCP・プラグイン最小化

使わない MCP は無効化する。起動時にシステムプロンプトへ注入されるためセッション全体のトークンに影響する。

## スキルを使う

スキル検索: https://skills.sh

| スキル | 用途 |
|---|---|
| `/init` | CLAUDE.md の初期生成 |
| `/genshijin` | 日本語出力の圧縮（約80%削減） |
| `/insights` | Claude Code セッション分析 |
| `/loop [間隔] [コマンド]` | 定期実行 |

### genshijin 強度

```
/genshijin          # 通常（デフォルト）
/genshijin 丁寧     # ビジネス向け
/genshijin 極限     # 最大圧縮
```

## モデルを使い分ける

> [!info] 判断基準
> 推論の複雑さでモデルを選ぶ。設計・デバッグは Opus、日常コーディングは Sonnet、単純な変換・確認は Haiku。コストは Haiku が最も安い。

| 用途 | モデル | ID |
|---|---|---|
| 複雑な設計・推論 | Opus 4.6 | `claude-opus-4-6` |
| 通常のコーディング | Sonnet 4.6（デフォルト） | `claude-sonnet-4-6` |
| 単純な変換・確認 | Haiku 4.5 | `claude-haiku-4-5-20251001` |

```
/model haiku    # 一時切り替え
```

## Hooks で自動化する

> [!info] なぜ Hooks か
> ツール実行・応答終了などのイベントに反応してシェルコマンドを自動実行する。「毎回手動で lint を実行する」「完了に気づけない」といった摩擦を排除できる。設定は `/update-config` スキルを使う。

`settings.json` の `hooks` フィールドで定義。

### イベント種別

| イベント | タイミング |
|---|---|
| `PreToolUse` | ツール実行前（lint、バックアップなど） |
| `PostToolUse` | ツール実行後（フォーマッター、テスト実行など） |
| `Stop` | Claude の応答終了時（デスクトップ通知など） |
| `Notification` | 通知発生時 |

### 設定例

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"完了\" with title \"Claude\"'"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "cd $CLAUDE_PROJECT_DIR && npx eslint --fix $CLAUDE_TOOL_INPUT_FILE_PATH 2>/dev/null || true"
          }
        ]
      }
    ]
  }
}
```

`matcher` はツール名の正規表現。空文字列は全ツールにマッチ。

## メモリシステムを使う

> [!info] なぜメモリか
> 会話コンテキストはセッションが終わるとリセットされる。メモリはセッションをまたいで自動ロードされるため、「また同じことを説明する」を防げる。

場所: `~/.claude/projects/<project>/memory/`

### メモリの種類

| タイプ | 用途 |
|---|---|
| `user` | ユーザーの役割・好み・知識レベル |
| `feedback` | 「こうしてほしい/こうしないで」の指示 |
| `project` | 進行中の作業・決定事項・背景 |
| `reference` | 外部リソースの場所（Linear, Grafana等） |

### 保存のタイミング

- 「覚えておいて」と明示的に言う
- Claude が作業を通じて判断・嗜好を学習したとき（自動）

### 保存されないもの

- コードパターン・アーキテクチャ（コードから読める）
- git 履歴・最近の変更（`git log` が正）
- セッション内の一時的な作業状態

### 構造

```
memory/
├── MEMORY.md          # インデックス（常にコンテキストに読み込まれる）
├── user_role.md
├── feedback_terse.md
└── project_context.md
```

## マルチエージェントを活用する

> [!info] なぜサブエージェントに委譲するか
> 広範な調査をメインコンテキストで行うと検索結果が文脈を汚染し、推論の質が下がる。サブエージェントはメインとは独立したコンテキストで動くため、メインをクリーンに保てる。

### エージェント種別と使い分け

| タイプ | 用途 | 特徴 |
|---|---|---|
| `Explore` | コードベース探索・ファイル検索 | 高速・読み取り専用 |
| `Plan` | 実装設計・アーキテクチャ検討 | 代替案の比較・トレードオフ評価 |
| `general-purpose` | 複合タスク・リサーチ | 全ツール使用可 |
| `claude-code-guide` | Claude Code/API の質問 | 専門知識あり |

### 並列 vs 直列

- **並列:** 独立したタスク（A ファイルの調査と B ファイルの調査）
- **直列:** 前の結果が次に必要（調査 → 設計 → 実装）

### フォアグラウンド vs バックグラウンド

- **フォアグラウンド（デフォルト）:** 結果が次のステップに必要
- **バックグラウンド:** 長時間タスクを並行して走らせる（完了時に通知）
