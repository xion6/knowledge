# プロンプト集

LLM のデフォルト挙動を変える、効果が高いプロンプトのリファレンス。

> 強いプロンプトはコンテンツではなく、**プロセスかフォーマット**を制約する。

---

## 思考強化

LLM の推論モードそのものを変えるプロンプト。

### 盲点・前提の挑戦

```
あなたの考えに潜む盲点を洗い出し、重要な前提を問い直し、
私が見落としているかもしれない別の視点を提示することで、
あなた自身の思考に挑戦してください
```

**なぜ効くか:** モデルを「検証の客体」にする。追従モードから批判的検討モードへ切り替わる。

---

### 語彙から発想する

```
〇〇をブラッシュアップしたいときの形容詞一覧を教えてください。
〇〇用語もお願いします
```

例: Webサイト → デザイン用語 / API設計 → アーキテクチャ用語

**なぜ効くか:** 出力の品詞を指定することで、「手順」ではなく「質の語彙」を引き出す。ブレストの素材として使いやすい。

---

### スティールマン（反論の強化）

```
私の主張に対して、最も強力な反論を作ってください。
私の立場を弱めるためではなく、私が見落としている本質的なリスクを特定するために。
```

---

### 前提の列挙

```
この判断が正しいために、何が真でなければならないか？
前提を5つ列挙してください。
```

---

## コード設計

### 設計問題の抽出

```
このコードの設計上の問題点を3つ挙げ、それぞれの修正案を提示してください。
バグではなく、保守性・拡張性・責務の観点で。
```

---

### リファクタリング前の境界確定

```
このコードをリファクタリングする前に、
変えてはいけない振る舞いを列挙してください。
```

**なぜ効くか:** 変更範囲を先に確定させる。「どう直すか」より「何を守るか」を先に考えさせる。

---

### API 設計レビュー

```
このAPIの設計を以下の観点で評価してください：
1. 将来の拡張性
2. 呼び出し元の認知負荷
3. エラーハンドリングの一貫性
問題があれば、具体的な修正案も示してください。
```

---

## デバッグ

### 仮説の発散

```
このエラーの原因として考えられる仮説を5つ挙げてください。
それぞれについて、最短の検証方法も示してください。
```

---

### 最小再現ケースの特定

```
このバグを再現する最小のコードを特定してください。
どの部分を取り除いても問題が消えないか、順番に考えてください。
```

---

## 言語化支援

### ノーコード説明（PR・ドキュメント）

```
この実装の意図を、コードを読んでいない人でも理解できるよう1段落で説明してください。
「何を」ではなく「なぜ」を中心に。
```

---

### PR 説明文

```
このPRの説明文を Why → What → How の順で書いてください。
Why: なぜこの変更が必要か
What: 何を変えたか
How: どう実装したか
```

---

### 仕様の構造化

```
以下の要件を整理してください：
- 曖昧な部分を明示する
- 未定義のエッジケースを列挙する
- 優先度をつける

要件: [ここに貼る]
```

---

## 発散・収束

### 制約なし発散

```
この問題の解決策を、実現可能性を無視して10個挙げてください。
絞り込みは次のステップで行います。
```

---

### 多軸評価による収束

```
上記の案を以下の3軸でランキングしてください：
1. 実装コスト（低・中・高）
2. 効果の即効性
3. 長期リスク

表形式で。
```

---

### 類似物からの発想

```
この問題に似た構造を持つ、全く別のドメインの事例を3つ挙げてください。
それぞれから学べる解決パターンを示してください。
```
