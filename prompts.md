# プロンプト集

LLM のデフォルト挙動を変える、効果が高いプロンプトのリファレンス。

> 強いプロンプトはコンテンツではなく、**プロセスかフォーマット**を制約する。

> [!note] 使用頻度マーク
> ★★★ = 高頻度（日常的に使う） / ★★ = 中頻度（局面で使う） / ★ = 低頻度（特定状況のみ）

## 批判的思考

### 盲点・前提の挑戦 ★★★

```
あなたの考えに潜む盲点を洗い出し、重要な前提を問い直し、
私が見落としているかもしれない別の視点を提示することで、
あなた自身の思考に挑戦してください
```

> [!info] なぜ効くか
> モデルを「検証の客体」にする。追従モードから批判的検討モードへ切り替わる。スティールマンとの違いは対象で、こちらは「自分の思考そのもの」への挑戦。スティールマンは「主張への反論」の構築。

### スティールマン（反論の強化） ★★

```
私の主張に対して、最も強力な反論を作ってください。
私の立場を弱めるためではなく、私が見落としている本質的なリスクを特定するために。
```

> [!info] なぜ効くか
> 「弱めるためではなく」という目的の再定義が核心。モデルに「敵ではなく盲点を見つける協力者」の役割を与えることで、形式的な批判ではなく本質的なリスクを引き出す。盲点・前提の挑戦との違いは対象で、こちらは「主張への最強の反論」を構築する。

### 前提の列挙 ★★

```
この判断が正しいために、何が真でなければならないか？
前提を5つ列挙してください。
```

> [!info] なぜ効くか
> 「判断が正しいために」という条件節が核心。「前提は何か」ではなく「これが成り立つために何が真でなければならないか」と問うことで、暗黙の前提を顕在化させる。5という数を指定することで、自明な前提を越えた深いレイヤーまで掘らせる。

## コード設計・デバッグ

### 設計問題の抽出 ★★★

```
このコードの設計上の問題点を3つ挙げ、それぞれの修正案を提示してください。
バグではなく、保守性・拡張性・責務の観点で。
```

> [!info] なぜ効くか
> 「バグではなく」という除外指定が鍵。デバッグモードではなく設計評価モードに誘導する。3つという数の制限で、枝葉ではなく本質的な問題に絞らせる。

### 仮説の発散 ★★★

```
このエラーの原因として考えられる仮説を5つ挙げてください。
それぞれについて、最短の検証方法も示してください。
```

> [!info] なぜ効くか
> 原因の特定を急がせずに発散させる。1つ目の仮説が間違いでも代替仮説がすでに手元にある。「最短の検証方法」を同時に要求することで、仮説が実験可能な形に整理される。

### リファクタリング前の境界確定 ★★

```
このコードをリファクタリングする前に、
変えてはいけない振る舞いを列挙してください。
```

> [!info] なぜ効くか
> 変更範囲を先に確定させる。「どう直すか」より「何を守るか」を先に考えさせる。

### API 設計レビュー ★★

```
このAPIの設計を以下の観点で評価してください：
1. 将来の拡張性
2. 呼び出し元の認知負荷
3. エラーハンドリングの一貫性
問題があれば、具体的な修正案も示してください。
```

> [!info] なぜ効くか
> 評価軸を事前に与えることで「全体的にどうか」ではなく構造的な評価を引き出す。「認知負荷」の軸は特に重要で、正しく動くコードと使いやすいAPIは別問題であることを明示している。

### 責務の過剰集中の特定 ★★

```
このコードで責務が過剰に集中しているクラス・関数・モジュールを特定してください。
その責務を分離するとしたら、どう切り出すか示してください。
分離のトレードオフ（複雑性の増加・オーバーエンジニアリングのリスク）も含めて。
```

> [!info] なぜ効くか
> 「分離せよ」ではなく「特定してから、トレードオフを含めて示せ」という順序が重要。モデルが自動的に分割を推奨するのを抑制し、現実的な判断を促す。

### テスト境界の設計 ★★

```
このコードに対してテストを書く前に、以下を整理してください：
1. テストすべき振る舞い（外部から観察可能なもの）
2. テストすべきでない実装の詳細
3. テストが困難な部分とその理由

テストが困難な部分については、設計上の問題がないかも指摘してください。
```

> [!info] なぜ効くか
> 「何をテストするか」より前に「何はテストしないか」を問うことで、実装詳細へのカップリングを防ぐ。テストの困難さを設計の問題として捉え直す視点が、コードの改善につながる。

### 最小再現ケースの特定 ★★

```
このバグを再現する最小構成を特定してください。
以下の手順で：
1. 現在の再現条件をすべて列挙する
2. 各条件を一つずつ取り除き、バグが消えるかを判定する
3. バグが消えない最小の条件セットを特定する

最終的に「これがなければ再現しない」という条件だけを残してください。
```

> [!info] なぜ効くか
> 「最小構成を見つけろ」という指示では曖昧。除外操作を順番に実行させるプロセス制約にすることで、モデルが体系的に絞り込む。「消えない」という否定形の判定基準が特に重要。

### 技術記事のレビュー ★★

````
# 目的
技術ブログ記事のドラフトを読み、技術的な正確性、コードの妥当性、最新のベストプラクティスとの乖離を厳格に検証してください。

# 記事内容
[ここに記事の本文(コード含む)を貼り付けてください]

# 検証タスク
1. **コードレビュー**: 構文エラー、論理的なバグ、非推奨(Deprecated)なメソッドの使用がないか確認してください。
2. **バージョン確認**: 2026年現在の最新安定版と比較し、古い情報(APIの変更など)がないか確認してください。
3. **用語の正確性**: 技術用語が正確に使われているか(例: 引数とパラメータの混同、プロトコルの名称ミスなど)を確認してください。
4. **セキュリティ・パフォーマンス**: セキュリティリスク(SQLインジェクション、ハードコードされた秘密情報など)や、著しく効率の悪い実装がないか指摘してください。

# 出力形式
以下の3セクションで構成してください。

### 1. 修正が必要な箇所(テーブル形式)
| 箇所 | 判定 | 理由・技術的根拠 | 修正後の案 |
| :--- | :--- | :--- | :--- |
| (コード or 文言) | (致命的/警告/推奨) | | |

### 2. 環境・バージョンに関する補足
(例: Node.js vXX以降では動作が異なる、等の注意点)

### 3. 全体的な信頼性スコア
(100点満点中何点か、および信頼性を高めるためのアドバイス)

# 注意事項
- 根拠となる公式ドキュメント名やRFCなどの規格名があれば適宜言及してください。
- 動作環境(OSや言語のバージョン)によって挙動が変わる場合は、その旨を明記してください。
````

> [!info] なぜ効くか
> 「レビューして」だけだと指摘が浅く分散する。「コード/バージョン/用語/セキュリティ」の4観点に分解することで、それぞれ別の専門家視点が起動し見落としが減る。判定を「致命的/警告/推奨」の3段階で強制することで、すべてが同じ重みで並ぶフラットな指摘リストを防ぎ、優先度が可視化される。信頼性スコアの要求は、個別指摘の総和ではなく「公開して大丈夫か」というメタ判断を引き出す装置として効く。

## 言語化

### PR 説明文 ★★★

```
このPRの説明文を Why → What → How の順で書いてください。
Why: なぜこの変更が必要か
What: 何を変えたか
How: どう実装したか
```

> [!info] なぜ効くか
> 順序の指定が核心。How から始めると実装の説明になるが、Why から始めることで変更の文脈と意図が前面に出る。レビュアーが「なぜこの変更か」を理解してからコードを読める状態を作る。

### ノーコード説明（PR・ドキュメント） ★★

```
この実装の意図を、コードを読んでいない人でも理解できるよう1段落で説明してください。
「何を」ではなく「なぜ」を中心に。
```

> [!info] なぜ効くか
> 受信者（コードを読んでいない人）の定義が抽象化レベルのフィルターになる。「なぜ」という制約で実装の詳細ではなく設計判断の言語化が促される。1段落という出力サイズの制限が、本質だけを絞り出させる。

### 仕様の構造化 ★★

```
以下の要件を整理してください：
- 曖昧な部分を明示する
- 未定義のエッジケースを列挙する
- 優先度をつける

要件: ［ここに貼る］
```

> [!info] なぜ効くか
> 「整理」ではなく「曖昧な部分・未定義・優先度」という3軸を指定することで、モデルが要件を受け入れるのではなく批判的に精査するモードに切り替わる。未定義のエッジケースの明示は、後工程での手戻りを防ぐ。

### 専門家向け再構成 ★

> 先行コンテキスト必須。情報を共有した後に使う変換プロンプト。

```
［テーマ］に関する重要な［会議／プレゼン／意思決定］を控えています。
これまでに共有した情報をもとに、この分野に非常に詳しく、時間に追われており、
表面的な分析に対して懐疑的な人物向けにその内容を書き直してください。
真の専門家が感心するような深い論点に絞り、当たり前の内容は排除し、
その人の知的水準と時間的制約に敬意を払う形で提示してください。
```

> [!info] なぜ効くか
> 受信者ペルソナを品質フィルターにしている。「深くしてください」と直接頼むより、「懐疑的な専門家が読む」と定義する方がLLMの出力が変わる。「当たり前の内容は排除」という除外指示が特に効く。

**使用例:**

| テーマ | 場面 | 想定読者（懐疑的な専門家） |
|---|---|---|
| マイクロサービス移行の提案 | CTO向けプレゼン | 過去に移行失敗を経験したCTO |
| 技術的負債の予算申請 | 経営会議 | ROIで判断する非技術系CFO |
| セキュリティインシデント報告 | 緊急意思決定 | 原因より対策を求めるVP |
| 新技術スタック導入の是非 | テックリード会議 | 学習コストを懸念するシニアエンジニア |
| パフォーマンス改善の成果報告 | ステークホルダー更新 | 数字の文脈を問うプロダクトマネージャー |

## 執筆・協働ライティング

### 批評 → 提案の2段構え ★★★

```
まず対象を読み手視点でレビューし、問題点を列挙してください。
その後で改善案を出して。分析と合成を分けて進めること
```

> [!info] なぜ効くか
> 「直して」とだけ頼まれると、モデルは分析を省略して即編集モードに入り、根本問題を見逃したまま表層を整える。批評を明示的に先行させると、診断と処方が分離され、読み違いが早期に露出する。依頼者も「モデルの読みの質」を検証でき、誤った前提のまま作業が進む事故が減る。

### 読み手の指定 ★★★

```
この文章を読むのは誰で、何秒で何を判断する人か。
その前提で、読み手の判断に寄与しない情報を削ぎ落とし、
最初の数秒で価値が伝わる構造で書いてください
```

> [!info] なぜ効くか
> 無指定だとモデルは「網羅的な正しさ」に最適化し、情報を盛る方向に引っ張られる。読み手を指定した瞬間、最適化軸が「読み手の時間と判断」に移り、削る判断が自動で働く。読み手が10秒で判断するなら、10秒を超えた情報を「無駄」と明確に判定できる。

### 素材の所在と取材の指定 ★★

```
書き始める前に [パス / 関連資料 / 既存文書] を読んでください。
不足情報があれば、書き始める前に質問で埋めて
```

> [!info] なぜ効くか
> 文脈不足のまま書き始めると、モデルは当たり障りのない一般論に逃げる。素材の所在を明示すると事実に根ざした具体性が出る。また「書く前に確認」の一言で、見切り発車を避けて欠けた情報を先に埋めに行く動きになる。これがないと、想像で埋めたフィクションが混ざる。

### ゼロベース許可 ★★

```
既存の構造・文章・設計に縛られず、ゼロベースで再構成してください。
現状を尊重するより、目的の達成を優先してください
```

> [!info] なぜ効くか
> 既存物を渡されるとモデルはデフォルトで「編集モード」に入り、前提そのものを疑わない。明示的な許可があって初めて「根本から再設計する」スイッチが入る。これがないと、筋の悪い前提の上に改善を積み上げてしまい、表層は整うが本質はズレたままの文章になる。

### 複数案の強制 ★★

```
いきなり完成版を書かず、軸の違う切り口を3案まず提示してください。
私が選んでから、1本化して書いて
```

> [!info] なぜ効くか
> 単一案を求めると、モデルは「最も無難な平均解」に収束する。特にゼロから書く自己表現系文書では、書き手の声や選好が反映されないまま完成版が出るため、修正コストが跳ね上がる。複数案を挟むと選択権が書き手に戻り、モデルの独断による凡庸化を防げる。意思決定系の [ビジネス戦略 3案 → ランク付け](#ビジネス戦略-3案--ランク付け) が「評価して順位をつける」のに対し、こちらは「書き手が選ぶ」ための発散である点が異なる。

### ドキュメント構成のリファクタ ★★

```
リポジトリ内のドキュメント（.md 群）の分割・命名・ディレクトリ構成を最適化してください。

着眼点:
- 重複している内容
- 1ファイルに責務が混在していないか（仕様 / 設計 / ロードマップ / 用語など）
- リポジトリ名と二重になる冗長なプレフィックス
- 同じトピックが複数ファイルに散らばっていないか

進め方:
1. 既存の .md を読んで診断
2. 提案の前に「分割粒度」「将来コードを置く場所」「プレフィックス方針」を確認
3. plan を提示して承認を得てから実装
4. リネームは `git mv`、重複は片方を残してリンクで誘導、各文書冒頭に関連文書リンク
5. 旧ファイル名の参照と内部リンク先の実在を grep で検証
```

> [!info] なぜ効くか
> 「整理して」だけだと表層的な見出し直しに終わる。着眼点を4軸（重複/責務混在/冗長プレフィックス/散逸）で先に与えることで診断モードに切り替わる。「提案の前に方針を確認」「planで承認」の二重ゲートで、合意なくファイルが動き出すのを防ぐ。`git mv` の指定で履歴を保ち、最後の grep 検証で内部リンク切れを未然に潰す。

## アイデア創出

### ビジネスモデルの構造移植 ★

```
以下の手順で進めてください。

【ステップ1: 構造の分解】
「［移植元のビジネスモデル名・事例］」が機能する理由を、以下の3点で説明してください：
1. 価値の核心（誰のどんなコストや摩擦を解消しているか）
2. 成立条件（このモデルが機能するために必要な環境・前提）
3. 収益の源泉（なぜ対価を払う人がいるか）

【ステップ2: 移植先の探索】
上記の構造を「［移植先のドメイン・業界・文脈］」に適用します。
- ステップ1の成立条件に対応する要素が移植先に存在するか確認する
- 存在する場合、具体的な事業アイデアを3つ提案する
- 存在しない場合、何が欠けているかを明示した上で、代替アイデアを示す
```

> [!info] なぜ効くか
> 「先に構造を分解してから移植する」というプロセス制約が核心。直接「移植先に当てはめて」と聞くと「Airbnb for X」式の表面的なアナロジーになる。成立条件を先に言語化させることで、移植先に同等の条件が揃っているかを自己検証する手順が生まれる。

**使用例（移植元 × 移植先の組み合わせ例）:**

| 移植元 | 移植先の例 | 問うべき成立条件 |
|---|---|---|
| サブスクリプション（Spotify） | 法律相談 / 栄養指導 | 繰り返しニーズがあるか、単発より継続の方が双方に価値があるか |
| マーケットプレイス（Airbnb） | 撮影スタジオ / 農機具 | 遊休資産の保有者と利用者が分離しているか |
| SaaS化（Salesforce） | 建設業の工程管理 / 農業の収穫管理 | 手動プロセスが標準化可能か、データ蓄積に価値があるか |
| フリーミアム（Notion） | 専門家向けツール / 業界特化ナレッジベース | 無料層が有料層の獲得コストを下げる構造があるか |

### 語彙から発想する ★

```
［テーマ］をブラッシュアップしたいときの形容詞一覧を教えてください。
［テーマ］用語もお願いします
```

例: Webサイト → デザイン用語 / API設計 → アーキテクチャ用語

> [!info] なぜ効くか
> 出力の品詞を指定することで、「手順」ではなく「質の語彙」を引き出す。ブレストの素材として使いやすい。

### 類似物からの発想 ★

```
この問題に似た構造を持つ、全く別のドメインの事例を3つ挙げてください。
それぞれから学べる解決パターンを示してください。
```

> [!info] なぜ効くか
> 「別のドメイン」という制約が既存の思考パターンからの離脱を強制する。同じ問題を別の抽象化レベルで見ることで、ドメイン固有の前提に縛られない解決策が出やすい。

## 意思決定

### 発散→収束（2ステップ） ★★

```
【ステップ1: 発散】
この問題の解決策を、実現可能性を無視して10個挙げてください。
重複や矛盾を恐れず、思考の幅を最大化してください。

【ステップ2: 収束】
上記の案を以下の3軸でランキングしてください：
1. 実装コスト（低・中・高）
2. 効果の即効性（短期／長期）
3. 長期リスク

表形式で示し、最終的に推奨案を1つ選んでその理由を述べてください。
```

> [!info] なぜ効くか
> 発散と収束を同一プロンプトに入れることで、モデルが最初から絞り込みを意識して発散が浅くなるのを防ぐ。ステップを明示的に分けることで、評価前の案に引っ張られるアンカリングを抑制する。「重複や矛盾を恐れず」という許可指示が発散の質を上げる。

### シナリオ比較 → 1結論（Tree of Thought） ★★

```
［テーマ］を5つのシナリオに分けて比較評価する。
各シナリオは以下の3ステップで分析する：
1. 前提条件と適用コンテキスト
2. 期待されるメリットとリスク
3. ［評価軸A］と［評価軸B］でのスコア

最後に、上記の分析に基づき最も根拠のある結論を1つ出す。
```

> [!info] なぜ効くか
> LLMが最初の答えに飛びつくのを構造で防ぐ。5分岐で発散させ、評価軸で収束させる Tree of Thought の手動実装。評価軸を自分で指定することで「根拠」が恣意的にならない。

**テーマ例:**

| テーマ | 評価軸A | 評価軸B |
|---|---|---|
| 認証方式の選定（JWT / Session / OAuth） | セキュリティ | 実装コスト |
| DBの選定（RDB / NoSQL / NewSQL） | スケーラビリティ | 運用負荷 |
| フロントエンドフレームワーク選択 | 学習コスト | 長期保守性 |
| マイクロサービス vs モノリス | 開発速度 | 障害影響範囲 |
| デプロイ戦略（Blue-Green / Canary / Rolling） | リリースリスク | ロールバック速度 |

### 戦略パターンの抽出 ★

> 先行コンテキスト必須。プロジェクト・ビジネスの状況を共有した後に使う。

```
最近の［テーマ／プロジェクト／ビジネス］に関するやり取りをもとに、
あなたはハイレベルな戦略アドバイザーとして振る舞ってください。
私が見落としている可能性のある重要なパターンや示唆を3つ挙げ、
それぞれが全体においてなぜ重要なのかを説明してください。
短期的な成果と長期的なポジショニングの両方を考慮に入れてください。
```

> [!info] なぜ効くか
> 「戦略アドバイザー」というロール指定で俯瞰視点を引き出す。「見落とし」を明示的に求めることで追従モードを抑制する。短期・長期の両軸を指定することで、どちらか一方に偏った示唆を防ぐ。

**使用例:**

| テーマ | 見落としやすいパターン例 |
|---|---|
| SaaSプロダクト開発 | チャーン率の先行指標を見ていない |
| 採用・チーム拡大 | 採用速度とオンボーディング品質のトレードオフ |
| 新規市場参入 | 初期トラクションが長期的な堀を妨げるケース |
| 技術的負債の解消 | 解消順序が開発速度に与える非線形な影響 |
| グロース施策 | 短期CVR改善が長期ブランドを毀損するリスク |

### ビジネス戦略 3案 → ランク付け ★

> 先行コンテキスト必須。ビジネス目標・課題を共有した後に使う。

```
私のビジネス目標と課題についてあなたが把握していることをもとに、
［具体的なビジネス課題］に対して3つの異なる戦略を提案してください。
それぞれの戦略について、主なメリット、想定されるデメリット、実行に必要な条件を含めてください。
3つすべての戦略を提示した後、それらを私にとって効果的な順にランク付けし、
その順位付けに影響した重要なトレードオフを説明してください。
```

> [!info] なぜ効くか
> 「全案を出してからランク付け」の順序が重要。先にランク付けさせると最初の案に引っ張られる。「実行に必要な条件」の要求で、アイデア止まりにならず実現可能性まで問える。

**課題の例:**

| ビジネス課題 | 「異なる」の軸を補足すると良い例 |
|---|---|
| プロダクトのマネタイズ方法を決めたい | 即収益 / 長期LTV / ユーザー摩擦の少なさ |
| 技術的負債の解消優先度 | リスク軽減 / 開発速度 / コスト |
| 新機能の開発を内製か外注か | スピード / 品質コントロール / コスト |
| ユーザー獲得チャネルの選定 | CAC / スケーラビリティ / ブランド適合 |
| チーム拡大のタイミングと方法 | 即戦力 / 文化フィット / 採用コスト |

### 成功チェックリストによる計画評価 ★

```
私は［計画の内容］についての計画を立てました。
この分野で成功しているアプローチに関する知見を踏まえ、
専門家レベルの成功チェックリスト（計画が成功するために満たすべき重要な条件のリスト）を作成し、
それを用いて私の計画を評価してください。
各チェック項目について、その項目がなぜ重要かを説明し、
私の計画がどの程度それに対応できているかを評価し、
必要に応じて具体的な改善案を提示してください。
成功または失敗に最も直結する要素に重点を置いてください。
```

> [!info] なぜ効くか
> 「ルーブリックを先に生成してから評価する」構造が核心。評価基準を後付けで作ると計画に合わせて逆算されるが、先に専門家基準を立てることで恣意的な評価を防ぐ。「失敗に直結する要素に重点」という優先度指定も重要。

**使用例:**

| 計画 | 失敗に直結しやすい要素 |
|---|---|
| SaaSプロダクトのローンチ計画 | ICP定義の曖昧さ、課金導線の未検証 |
| 技術スタック移行計画 | データ移行のロールバック手順、並行稼働期間 |
| 新機能リリース計画 | フィーチャーフラグ設計、監視・アラート整備 |
| 採用計画 | 評価基準の一貫性、カルチャーフィット検証方法 |
| グロース施策の実行計画 | 計測設計（何をもって成功とするか） |

## ナレッジ管理

### 合意事項の記録・永続化 ★★

```
直前のやり取りで合意した方針・観点を、次回以降の作業で自動的に参照されるよう記録して。

- 適切な保存先（CLAUDE.md / README.md / memory / その他）を選び、理由を1行で示す
- 既存記述と重複・矛盾があれば整理する
- 「観点（=増えていくナレッジ）」と「プロセス（=作業手順）」は分けて記録する
- 記録後、何がトリガーで参照されるかを1行で明示する
```

> [!info] なぜ効くか
> 「記録して」だけだと保存先の選択・既存との整合・参照条件がすべて暗黙になる。保存先の選定理由を明示させることで、どこに何があるかの構造が維持される。「観点とプロセスを分ける」指示はナレッジの腐りやすさの違いを反映しており、手順が変わっても観点は残る設計にする。「参照トリガーの明示」は、記録が実際に使われるかの事後確認を自動化する。
